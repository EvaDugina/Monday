Три замечания пользователя по приложению MONDAY:

1. «Блок с погодой не закрепляй сверху при скролле».
2. «Проблема (см скриншот): Элементы управления картинками при редактировании фона располагаются не в тех местах фотографии» (скриншот: рамка выделения, кнопка удаления, ручки ресайза и подпись «180 x 135» смещены относительно видимого изображения).
3. «Синхронизируй фон и всю информацию по аккаунту а не по localStorage. Сейчас в разных браузерах разная информация. Пользователь будет один, но я хочу синхронизацию между браузерами».

**Согласованные решения (апрув «Да, фиксирую»):**
- Изображения фона — отдельное серверное blob-хранилище; в синхронизируемый снапшот кладём только ссылки + геометрию (не base64).
- Объём синхронизации по аккаунту: фон + город погоды + категории + задачи + архив (категории/задачи/архив уже синхронизируются).

## Функциональные требования

- **Скролл блока погоды:** Система должна прокручивать блок погоды вместе с содержимым страницы и не закреплять его у верхнего края вьюпорта.
  - **Пояснение:** Сейчас блок зафиксирован (`position: fixed`) и остаётся на месте при скролле; пользователь хочет, чтобы он уезжал вместе со страницей.
  - **Источник:** "Блок с погодой не закрепляй сверху при скролле".
  - **Acceptance hint:** При скролле вниз блок погоды уходит вверх вместе с контентом, а не остаётся на экране.

- **Позиционирование контролов редактора фона:** Система должна располагать элементы управления изображением (кнопка удаления, 4 ручки изменения размера, подпись размера, рамка выделения) точно по краям/углам видимого изображения при любых пропорциях и размере.
  - **Пояснение:** Сейчас контролы привязаны к декларированному боксу кадра, который из-за `object-fit: contain` и высоты-по-умолчанию не совпадает с реально отрисованной картинкой.
  - **Источник:** "Элементы управления картинками при редактировании фона располагаются не в тех местах фотографии".
  - **Acceptance hint:** Кнопка удаления в правом-верхнем углу картинки, ручки в её 4 углах, подпись под ней, рамка обводит саму картинку.

- **Кросс-браузерная синхронизация фона:** Система должна хранить фон (изображения + позиция/размер/поворот/прозрачность) на стороне аккаунта и отдавать одинаковый фон во всех браузерах одного пользователя.
  - **Пояснение:** Сейчас фон лежит только в `localStorage`, поэтому в разных браузерах он разный.
  - **Источник:** "Синхронизируй фон ... а не по localStorage. Сейчас в разных браузерах разная информация".
  - **Acceptance hint:** Изменение фона в браузере A после синка/перезагрузки видно в браузере B.

- **Синхронизация настроек аккаунта:** Система должна синхронизировать через аккаунт все пользовательские настройки, которые сейчас лежат только в `localStorage` (как минимум фон и выбранный город погоды), совместно с уже синхронизируемыми задачами/категориями/архивом.
  - **Пояснение:** Пользователь хочет одинаковую «информацию по аккаунту» во всех браузерах.
  - **Источник:** "всю информацию по аккаунту а не по localStorage".
  - **Acceptance hint:** Выбранный город погоды одинаков во всех браузерах пользователя.

## Нефункциональные требования

- **Local-first не ломать:** При недоступности API приложение должно продолжать работать офлайн (TD-002), не теряя фон/настройки.
  - **Пояснение:** Существующая офлайн-устойчивость не должна регрессировать.
  - **Источник:** производное от требования синхронизации + текущая архитектура.
  - **Acceptance hint:** С уже загруженным фоном при остановленном API доска работает, статус `offline`, существующий фон рендерится.

- **Единая модель синхронизации:** Фон и настройки должны использовать ту же versioned-snapshot модель и разрешение конфликтов, что и задачи (TD-001), без серверного silent-merge.
  - **Пояснение:** Не вводить второй механизм синка; конфликты разрешает клиент.
  - **Источник:** производное от «синхронизации по аккаунту» + текущая модель.
  - **Acceptance hint:** Одновременная правка фона и задачи в двух браузерах не рушит синк; оба изменения выживают.

- **Лимиты полезной нагрузки:** Решение по фону должно учитывать лимит тела JSON-запроса и размер изображений — байты картинок не должны раздувать синхронизируемый снапшот и бэкапы.
  - **Пояснение:** Base64-картинки (до ~1.6MB × 6) не помещаются в JSON-снапшот разумного размера; поэтому отдельное blob-хранилище.
  - **Источник:** согласованное решение по хранению.
  - **Acceptance hint:** `PUT /api/tasks` несёт только ссылки+геометрию, без base64; картинки грузятся отдельным эндпоинтом.

- **Сохранность при миграции:** Существующие локальные фон/настройки при первом запуске новой версии не должны теряться — их нужно перенести/засидить в аккаунт по аналогии с seed задач.
  - **Пояснение:** У пользователя уже есть фон/город в localStorage; апгрейд не должен их обнулять.
  - **Источник:** производное от смены места хранения.
  - **Acceptance hint:** После апгрейда с локальным фоном он появляется на сервере и виден во втором браузере.

## Проблемы и замечания, превращённые в требования

- **Погода при скролле:** Сейчас блок погоды закреплён и остаётся на месте при прокрутке. Требуемое поведение: прокручивается вместе со страницей.
  - **Источник:** "Блок с погодой не закрепляй сверху при скролле".
  - **Acceptance hint:** Блок уезжает вверх при скролле.

- **Контролы фона:** Сейчас контролы редактора смещены относительно картинки. Требуемое поведение: контролы точно на краях видимого изображения.
  - **Источник:** скриншот + "располагаются не в тех местах фотографии".
  - **Acceptance hint:** bbox рамки совпадает с bbox картинки; контролы на её краях.

- **Рассинхрон между браузерами:** Сейчас фон и город погоды разные в разных браузерах (только `localStorage`). Требуемое поведение: одинаковые во всех браузерах через аккаунт.
  - **Источник:** "Сейчас в разных браузерах разная информация ... хочу синхронизацию между браузерами".
  - **Acceptance hint:** Фон/город, заданные в браузере A, появляются в браузере B.

## Открытые вопросы

- (Закрыто) Как синхронизировать тяжёлые изображения фона при лимите тела запроса — **решено:** отдельное серверное blob-хранилище + ссылки в снапшоте.
- (Закрыто) Что считать «информацией по аккаунту» — **решено:** фон + город + категории + задачи + архив.
- Нужно ли отдельно версионировать/бэкапить настройки — по умолчанию settings едут внутри `state.tasks_json`, поэтому попадают в существующие `task_backups` автоматически (байты картинок в бэкап не входят — задокументировать как известное ограничение).

## Не включено / требует уточнения

- Точный способ починки контролов фона (нормализация высоты под натуральное соотношение сторон против пересчёта бокса под `object-fit`) — ведущая гипотеза зафиксирована в плане; подтверждается эмпирически на живом приложении при реализации.
- Расширяемый общий объект настроек «на будущее» (тема, настройки доски/карточек) — в рамках этого фикса вводим `settings` только под фон и город; дальнейшие поля добавляются отдельно.
